# ListenRoom — Version 1.0.0

**Date:** 2026-04-14
**Status:** Current working build

---

## What This App Is

ListenRoom is a single-room synchronized music listening app for a small group of friends. There is no login, no database, and no playback controls. Music flows like a radio. Anyone can add songs. The queue is fair — it rotates between users so no one person monopolizes the playlist.

## Shared Types (`libs/shared/src/index.ts`)

### `QueueItem`

```ts
interface QueueItem {
  id: string        // UUID v4, generated at enqueue time
  title: string     // extracted by yt-dlp
  duration: number  // length in seconds (integer), default 180 if yt-dlp returns 0
  fileId: string    // first 12 hex chars of MD5(sourceUrl) — maps to /audio/{fileId}.mp3
  sourceUrl: string // original URL or "ytsearch1:..." search query
  addedBy: string   // display name of the user who added it
}
```

### `RoomState`

```ts
interface RoomState {
  currentSong: QueueItem | null   // song playing right now, or null
  startedAt: number | null        // Date.now() when this song started, or null
  queue: QueueItem[]              // round-robin preview — one entry per active user (their next song)
  userQueues: Record<string, QueueItem[]>  // full personal backlog, keyed by username
}
```

The `queue` field is *derived* — it is rebuilt from `userQueues` and the rotation order every time state changes. It shows what comes next one slot per user, not a flat list of every queued song.

### `RoomStateWithElapsed`

```ts
interface RoomStateWithElapsed extends RoomState {
  elapsed: number | null  // seconds since currentSong started, computed at read time
}
```

Sent only once, when a client emits `joinRoom`. The client uses `elapsed` to seek the audio element to the right position.

### `DownloadStatus`

```ts
type DownloadStatusType = 'pending' | 'downloading' | 'done' | 'error'

interface DownloadStatus {
  url: string
  status: DownloadStatusType
  progress?: number    // 0–100, present during 'downloading'
  message?: string     // error message, present on 'error'
  item?: QueueItem     // the completed item, present on 'done'
}
```

### `EVENTS`

```ts
const EVENTS = {
  // Client → Server
  JOIN_ROOM:          'joinRoom',
  ADD_TO_QUEUE:       'addToQueue',
  SKIP_SONG:          'skipSong',
  REMOVE_FROM_QUEUE:  'removeFromQueue',
  MOVE_TO_TOP:        'moveToTop',
  MOVE_TO_BOTTOM:     'moveToBottom',

  // Server → Client
  ROOM_STATE:         'roomState',
  SONG_STARTED:       'songStarted',
  QUEUE_UPDATED:      'queueUpdated',
  DOWNLOAD_STATUS:    'downloadStatus',
}
```

### Payload types

```ts
interface JoinRoomPayload      { username: string }
interface AddToQueuePayload    { url: string; username: string }
interface SkipSongPayload      { username: string }
interface QueueMutationPayload { songId: string; username: string }
```

## Backend Logic

### `main.ts`

- Custom `CorsIoAdapter` sets Socket.io `cors: { origin: '*' }` so any origin can connect.
- `app.enableCors({ origin: '*' })` for HTTP endpoints.
- Listens on port 4000.

### `AppModule`

Imports: `EventEmitterModule.forRoot()`, `AudioModule`, `RoomModule`, `QueueModule`. No HTTP controllers at the app level — routing is all through the gateway and module-level controllers.

---

### `QueueService` (`apps/api/src/app/queue/queue.service.ts`)

Turns a URL or search query into a `QueueItem` on disk.

**Paths (configurable via env vars):**
- `CACHE_DIR = process.env.AUDIO_CACHE_DIR ?? apps/api/audio-cache`
- `META_CACHE_PATH = process.env.STATE_DIR ? {STATE_DIR}/meta-cache.json : apps/api/.dev-state/meta-cache.json`

**`onModuleInit()`**
Creates `CACHE_DIR` if missing. Loads `metaCache` from `META_CACHE_PATH` if it exists.

**`downloadAndEnqueue(sourceUrl, addedBy, onProgress?)`**

1. Compute `fileId = MD5(sourceUrl).slice(0, 12)`
2. If `{fileId}.mp3` exists on disk AND `metaCache` has it → return cached `QueueItem` immediately (no yt-dlp call)
3. Otherwise spawn yt-dlp:
   - `--extract-audio --audio-format mp3 --audio-quality 0`
   - `--output {CACHE_DIR}/{fileId}.%(ext)s`
   - `--print after_move:%(title)s|||%(duration)s`
   - `--no-playlist`
   - If `/app/cookies.txt` exists → `--cookies /app/cookies.txt` (for age-gated videos in Docker)
4. Parse progress percentages from stdout/stderr, call `onProgress(pct)`
5. On exit code 0: split `|||` to get title/duration, default duration to 180 if missing
6. Save metadata to `metaCache` and flush `meta-cache.json`
7. Return `QueueItem`

**Error handling:**
- `ENOENT` spawn error → "yt-dlp is not installed or not found in PATH"
- Non-zero exit → last line of stderr as message
- MP3 not produced after exit 0 → "yt-dlp finished but mp3 was not produced — is ffmpeg installed?"
- Partial files (`.webm`, `.m4a`, `.opus`, `.ogg`, `.part`) cleaned up on failure

**`scheduleCleanup(fileId, isStillNeeded)`**
After 30 seconds, checks `isStillNeeded()` — if false, deletes the MP3 and removes from `metaCache`. Called by `RoomService` after every song advance. `isStillNeeded` returns true if the fileId appears in any user's personal queue.

### `RoomService` (`apps/api/src/app/room/room.service.ts`)

The single source of truth for room state. One instance per server lifetime.

**Internal state:**
```ts
private userQueues: Map<string, QueueItem[]>  // personal queues, username → songs
private userOrder: string[]                   // round-robin rotation order
private state: RoomState                      // { currentSong, startedAt, queue, userQueues }
private advanceTimer: NodeJS.Timeout | null
private saveTimer: NodeJS.Timeout | null
```

**`getRoomState()`**
Returns `state` plus computed `elapsed = (Date.now() - startedAt) / 1000`. Read-only.

**`enqueue(item)`**
1. Append to `userQueues[username]`
2. Add `username` to `userOrder` if not present
3. `rebuildPublicState()` — sets `state.queue` (one item per user from their front) and `state.userQueues` (full copy of all personal queues)
4. If `currentSong` is null → call `advanceSong()` immediately
5. Else → `scheduleSnapshot()`

**`advanceSong()`**
1. Clear pending timer
2. If `userOrder` is empty → clear `currentSong`/`startedAt`, emit `room.songAdvanced`, save snapshot, return
3. Shift first username from `userOrder`, shift their first song from `userQueues[username]`
4. If user still has songs → push username back to end of `userOrder`; else delete from `userQueues`
5. Set `currentSong = next`, `startedAt = Date.now()`, `rebuildPublicState()`
6. Call `queueService.scheduleCleanup(prevSong.fileId, isStillNeeded)` for the finished song
7. Emit `room.songAdvanced` via `EventEmitter2`
8. `scheduleSnapshot()`
9. `setTimeout(advanceSong, duration * 1000)`

**`removeFromQueue(username, songId)`**
Removes from personal queue, removes from rotation if now empty. Broadcasts via gateway after return.

**`moveToTop(username, songId)` / `moveToBottom(username, songId)`**
Reorders within the personal queue. `rebuildPublicState()` updates the shared `queue` preview.

**State persistence:**
- `scheduleSnapshot()` — debounced 500ms write to disk
- `flushSnapshot()` — immediate sync write (called on shutdown)
- `loadSnapshot()` — on `onModuleInit()`, reads JSON, restores userQueues + userOrder + state, re-arms advance timer

### `RoomGateway` (`apps/api/src/app/room/room.gateway.ts`)

Socket.io event handlers. No state of its own — delegates entirely to `RoomService` and `QueueService`.

**`handleJoin(data, client)`**
Logs the join, calls `getRoomState()`, emits `roomState` to the connecting client only.

**`handleAddToQueue(data, client)`**
1. Broadcast `downloadStatus { downloading, 0% }` to all clients
2. Await `queueService.downloadAndEnqueue(url, addedBy, progressCallback)`
3. On success: broadcast `downloadStatus { done }`, call `roomService.enqueue(item)`, broadcast `queueUpdated`
4. On error: emit `downloadStatus { error }` to submitting client only

**`handleSkip(data)`**
Gets current song. Returns early if `current.addedBy !== data.username`. Otherwise calls `advanceSong()`. The resulting `room.songAdvanced` event triggers `handleSongAdvanced` which broadcasts.

**`handleRemove(data)` / `handleMoveTop(data)` / `handleMoveBottom(data)`**
Call the corresponding `RoomService` method. If it returns `true`, broadcast `queueUpdated`.

**`handleSongAdvanced(state)`** (`@OnEvent('room.songAdvanced')`)
Broadcasts `songStarted` to all connected clients.

---

### `AudioController` (`apps/api/src/app/audio/audio.controller.ts`)

`GET /audio/:fileId.mp3`

- 404 if file not found
- Sets `X-Accel-Buffering: no` — tells Caddy not to buffer the response (prevents timeout on long files)
- Sets `Accept-Ranges: bytes`
- Supports `Range: bytes=X-Y` headers with HTTP 206 partial content
- Streams via `fs.createReadStream({ start, end })`

### `RoomController` (`apps/api/src/app/room/room.controller.ts`)

`GET /room` — returns current `RoomState` as JSON. Debug only.

## Frontend Logic

### `lib/socket.ts`

Singleton Socket.io client. Created once, reused across all hooks.

```ts
const url = isLocal ? 'http://localhost:4000' : window.location.origin
```

In production, `window.location.origin` is used because Caddy proxies `/socket.io*` on the same origin. In local dev, it points directly to port 4000.

---

### `hooks/useSocket.ts`

Manages the socket connection lifecycle.

- Two `useEffect`s to handle the race between socket connection and username availability:
  1. Attaches `connect`/`disconnect` handlers; emits `joinRoom` on connect if username is already set
  2. Watches `username` — emits `joinRoom` if socket is already connected when username first becomes available

---

### `hooks/useRoomState.ts`

Holds all React state for the room. Subscribes to socket events.

**State:** `currentSong`, `queue`, `userQueues`, `downloadStatuses`

**Event handlers:**
- `roomState` / `songStarted` / `queueUpdated` → call `applyState(state)` which sets all three state fields
- `downloadStatus` → upsert into `downloadStatuses` by `url`; schedule removal after 3s on `done`/`error`

**Actions:**
- `addToQueue(url)` — emits `addToQueue` with `{ url, username }`
- `skipSong()` — emits `skipSong` with `{ username }`
- `removeFromQueue(songId)` — emits `removeFromQueue`
- `moveToTop(songId)` / `moveToBottom(songId)` — emits respective events

---

### `hooks/useAudioPlayer.ts`

Manages the `<audio>` element. Does not hold React-visible state except `currentTime` (for the progress bar).

**Refs:**
- `audioRef` — the `<HTMLAudioElement>` itself
- `stallRetriesRef` — count of stall retry attempts (resets to 0 on successful play, max 3)
- `pendingSyncRef` — `{ elapsed, capturedAt }` stored when autoplay is blocked so the seek can compensate for elapsed wall-clock time

**`onRoomState`:**
1. Skip if audio is already playing the same src (prevents re-sync on reconnect if nothing changed)
2. Set `audio.src = /audio/{fileId}.mp3`, wait for `loadedmetadata`
3. Seek to `elapsed`, set `pendingSyncRef`, call `tryPlay()`

**`onSongStarted`:**
Set `audio.src`, seek to 0, call `tryPlay()`. If no song → pause and clear src.

**`tryPlay()`:**
Calls `audio.play()`. If rejected (autoplay blocked), attaches click/keydown listeners that call `resumeOnInteraction`.

**`resumeOnInteraction`:**
Adjusts seek by `elapsed + (Date.now() - capturedAt) / 1000` before playing (compensates for time since roomState arrived).

**`recoverAudio()`:**
On stall or error: saves `currentTime`, reloads `audio.src`, seeks back, plays. Up to 3 attempts.

---

### `hooks/useRoom.ts`

Thin composer — calls `useSocket`, `useAudioPlayer`, `useRoomState`, spreads all their returns into one object for `page.tsx`.

### Components

**`UsernameModal`**
Shown when `username === null`. Renders a centered modal with a text input (max 32 chars). On submit, calls `onConfirm(name)` which saves to localStorage.

**`NowPlaying`**
- Animated 4-bar wave (CSS `@keyframes wave`) in violet when playing
- Song title with `@keyframes marquee` scroll animation
- "Added by {username}" subtitle
- Progress bar (read-only, width = `currentTime / duration * 100%`, updates every ~250ms via `timeupdate`)
- Skip button: only rendered when `username === currentSong.addedBy`

**`Queue`** (Up Next)
Shows the round-robin preview: one song per active user, ordered by their turn. Displays the `addedBy` username (not the song title) — this is intentional, it shows whose slot is next, not a full tracklist.

**`PersonalQueue`** (My Queue)
Shows the current user's full backlog (song titles). Per-song controls:
- `↑` (move to top) — disabled on first item
- `↓` (move to bottom) — disabled on last item
- `✕` (remove)

**`SearchBox`**
Text input. On submit: if input starts with `http` → send as-is; else prefix `ytsearch1:`. Shows pulsing violet dot while any download is active, red error messages for failures.

### `app/page.tsx`

On mount: reads `localStorage["listenroom_username"]`. If missing, shows `UsernameModal`. If present, calls `useRoom(username)`.

Layout (dark zinc-950 background, max-w-7xl, flex):

- Left column (flex-[3]):
  - `NowPlaying`

  - Bottom row: `PersonalQueue` (flex-1) + `Queue` (flex-[1.4])

## Deployment

### Local dev (Docker)

`docker-compose.yml` runs four services:

| Service | Image | Purpose |
|---|---|---|
| `api` | built from `apps/api/Dockerfile` | NestJS backend |
| `web` | built from `apps/web/Dockerfile` | Next.js frontend |
| `caddy` | `caddy:2-alpine` | Reverse proxy on :80 |
| `cloudflared` | `cloudflare/cloudflared:latest` | Quick tunnel (no domain needed) |

Caddy routes:
- `/socket.io*` → `api:4000`
- `/api/*` → `api:4000`
- `/audio/*` → `api:4000`
- everything else → `web:3000`

`cloudflared` opens a quick Cloudflare Tunnel pointing at `caddy:80`. The URL appears in `docker compose logs cloudflared` — it looks like `https://random-words.trycloudflare.com`. Anyone with that link can access the room.

The `audio-cache` named volume persists MP3 files across container restarts.

### Production (pre-built images)

`docker-compose.prod.yml` uses pre-built images from Docker Hub:
- `${DOCKERHUB_USER}/listenroom-api:latest`
- `${DOCKERHUB_USER}/listenroom-web:latest`

Exposes ports 3000 and 4000 directly (no Caddy — intended for use behind an existing reverse proxy or directly on the VPS).

Mounts `./cookies.txt:/app/cookies.txt` for YouTube authentication if needed.

## Key Design Decisions

**Round-robin instead of FIFO**
A flat FIFO queue means a single user who adds 10 songs blocks everyone else for 10 songs. Round-robin ensures everyone gets a turn. The `userQueues` + `userOrder` structure implements this with O(1) advance.

**`addedBy` is a display name, not a socket ID**
Socket IDs change on every reconnect. A display name persists for the session (stored in localStorage). This is what identifies who "owns" a song for skip and queue mutation permissions.

**Snapshot persistence instead of "no state on restart"**
The original design reset all state on restart. The snapshot system means the queue survives a crash or redeploy. It's a JSON file, not a database — still no schema, no migrations, still acceptable to lose on a corrupted write.

**Custom `AudioController` instead of `@nestjs/serve-static`**
`serve-static` handles range requests but doesn't support `X-Accel-Buffering: no`. Without that header, Caddy buffers the entire MP3 before forwarding, killing long-file streams. The custom controller sets that header explicitly.

**Socket singleton in `lib/socket.ts`**
React 18 strict mode mounts/unmounts effects twice in dev. A new socket on every mount would cause double-join events. The singleton ensures one connection regardless of how many times effects fire.

**URL detection for socket endpoint**
`window.location.hostname === 'localhost'` → connect to `http://localhost:4000`; otherwise use `window.location.origin`. In production behind Caddy, the socket connects on the same origin as the page, and Caddy proxies it to the backend. This means no hardcoded hostnames in the frontend build.
