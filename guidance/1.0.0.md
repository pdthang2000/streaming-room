# ListenRoom — Version 1.0.0

## What Is This App

ListenRoom is a synchronized music listening room for a small group of friends.
There is one room, no login, no database. Anyone who opens the URL hears the same
song at the same point in time — like a radio station you can add songs to.

## Shared Types (`libs/shared/src/index.ts`)

Both the frontend and backend import exclusively from `@listenroom/shared`.
Types are never duplicated in either app.

### `QueueItem`

```ts
interface QueueItem {
  id: string        // random UUID, generated at enqueue time
  title: string     // song title extracted by yt-dlp
  duration: number  // length in seconds (integer), default 180 if missing
  fileId: string    // first 12 chars of MD5 hash of sourceUrl
                    // used to build the audio URL: /audio/{fileId}.mp3
  sourceUrl: string // original URL or ytsearch1:... query string
  addedBy: string   // socket.id of the client who added it
}
```

`fileId` is a hash of the source URL so that the same song URL always
maps to the same filename. If a file already exists on disk, we skip
re-downloading it.

### `RoomState`

```ts
interface RoomState {
  currentSong: QueueItem | null  // the song playing right now, or null
  startedAt: number | null       // Date.now() when this song started, or null
  queue: QueueItem[]             // ordered list of songs waiting to play
}
```

This is the complete state of the room. It lives entirely in memory on the
server. If the server restarts, it resets. That is intentional — no database.

### `RoomStateWithElapsed`

```ts
interface RoomStateWithElapsed extends RoomState {
  elapsed: number | null  // seconds since currentSong started, or null
}
```

An extension of `RoomState` that the server only sends once — when a client
first joins. The `elapsed` field is computed server-side as:

```ts
elapsed = (Date.now() - startedAt) / 1000
```

The client uses this to seek the audio element to the right position,
achieving sync without any peer-to-peer coordination.

### `DownloadStatus`

```ts
type DownloadStatusType = 'pending' | 'downloading' | 'done' | 'error'

interface DownloadStatus {
  url: string              // the sourceUrl that was submitted
  status: DownloadStatusType
  progress?: number        // 0–100, only present during 'downloading'
  message?: string         // human-readable error, only present on 'error'
  item?: QueueItem         // the completed item, only present on 'done'
}
```

The server broadcasts one of these every time the download state changes
for a song being added. The frontend uses it to show a per-song status indicator.

### `EVENTS`

```ts
const EVENTS = {
  JOIN_ROOM:       'joinRoom',       // client → server
  ADD_TO_QUEUE:    'addToQueue',     // client → server
  ROOM_STATE:      'roomState',      // server → joining client only
  SONG_STARTED:    'songStarted',    // server → all clients
  QUEUE_UPDATED:   'queueUpdated',   // server → all clients
  DOWNLOAD_STATUS: 'downloadStatus', // server → all clients
}
```

These are the only strings used for Socket.io event names. They are never
hardcoded as raw strings in application code — always reference `EVENTS.X`.

## Backend Logic

### `QueueService` (`apps/api/src/app/queue/queue.service.ts`)

Responsible for turning a URL or search query into a `QueueItem` on disk.

**`downloadAndEnqueue(sourceUrl, addedBy, onProgress?)`**

1. Compute `fileId = MD5(sourceUrl).slice(0, 12)`
2. Check if `audio-cache/{fileId}.mp3` already exists AND is in the in-memory
   `metaCache` map. If yes, skip download and return a new `QueueItem` with
   the cached title/duration.
3. Otherwise, spawn `yt-dlp` as a child process with these flags:
   - `--extract-audio --audio-format mp3` — audio only, no video
   - `--audio-quality 0` — best quality
   - `--output audio-cache/{fileId}.%(ext)s` — save with fileId as name
   - `--print after_move:%(title)s|||%(duration)s` — print metadata after done
   - `--no-playlist` — never download a full playlist
4. Parse progress percentage from stdout/stderr lines (e.g. `"42.3%"`)
   and call `onProgress(42.3)` so the gateway can broadcast it.
5. When the process exits with code 0, split the printed metadata on `|||`
   to get title and duration. Default duration to 180 if missing or NaN.
6. Store title/duration in `metaCache`, return the complete `QueueItem`.

**Error handling:**
- If `yt-dlp` binary is not found (`ENOENT`), throws a clear message.
- If yt-dlp exits non-zero (private video, geo-block, etc.), throws with the code.

**`onModuleInit()`**
Creates `audio-cache/` directory on startup if it does not exist.

---

### `RoomService` (`apps/api/src/app/room/room.service.ts`)

Owns the single source of truth for room state. There is one instance of
this service for the lifetime of the server process.

**State shape:**
```ts
{
  currentSong: QueueItem | null
  startedAt:   number | null
  queue:       QueueItem[]
}
```

**`getRoomState()`**
Returns the current state plus a computed `elapsed` value. Called when a
client joins so they can seek to the right position.

**`enqueue(item)`**
Pushes an item onto `queue`. If nothing is currently playing (`currentSong`
is null), immediately calls `advanceSong()`.

**`advanceSong()`**
The core playback engine:
1. Clears any pending advance timer.
2. If the queue is empty, sets `currentSong` and `startedAt` to null and returns.
3. Pops the first item from `queue`, sets it as `currentSong`, records
   `startedAt = Date.now()`.
4. Emits `room.songAdvanced` via `EventEmitter2` so the gateway can broadcast
   to all clients.
5. Schedules `setTimeout(advanceSong, duration * 1000)` to auto-advance
   when the song ends.

`RoomService` never talks to the gateway directly. It communicates through
`EventEmitter2` to keep the two decoupled.

---

### `RoomGateway` (`apps/api/src/app/room/room.gateway.ts`)

The Socket.io layer. Handles incoming client messages and outgoing broadcasts.

**`@SubscribeMessage('joinRoom')`**
Called when a client connects and emits `joinRoom`.
Responds to that client only (not broadcast) with `roomState` — the current
state including `elapsed`.

**`@SubscribeMessage('addToQueue')`**
Called when a client submits a song:
1. Immediately broadcasts `downloadStatus { status: 'downloading', progress: 0 }`
   to all clients so everyone sees something is happening.
2. Calls `QueueService.downloadAndEnqueue()`, passing a progress callback
   that broadcasts `downloadStatus` updates as yt-dlp runs.
3. On success: broadcasts `downloadStatus { status: 'done' }`, calls
   `RoomService.enqueue(item)`, broadcasts `queueUpdated` with new state.
4. On error: emits `downloadStatus { status: 'error', message }` back to
   the submitting client only.

**`@OnEvent('room.songAdvanced')`**
Listens for the internal `EventEmitter2` event fired by `RoomService`.
Broadcasts `songStarted` with the new state to all connected clients.

---

### `AudioModule` (`apps/api/src/app/audio/audio.module.ts`)

Uses `@nestjs/serve-static` to mount the `audio-cache/` directory at the
`/audio` route. This means `audio-cache/abc123def456.mp3` is accessible at
`http://localhost:4000/audio/abc123def456.mp3`.

`@nestjs/serve-static` handles HTTP Range requests automatically.
Range requests are essential — browsers send `Range: bytes=X-Y` headers when
an `<audio>` element seeks. Without range support, seeking does not work.

---

### `AppModule` (`apps/api/src/app/app.module.ts`)

Wires everything together:
```ts
imports: [
  EventEmitterModule.forRoot(),   // enables @OnEvent decorator
  AudioModule,                    // static file serving
  RoomModule,                     // room state + socket gateway
  QueueModule,                    // yt-dlp download logic
]
```

There are no controllers or HTTP routes. All client communication goes
through Socket.io (the gateway) or static file serving (audio).

## Frontend Logic

### `hooks/useRoom.ts`

This is the most important file on the frontend. Every piece of real-time
state flows through here.

**Refs (not state — do not trigger re-renders):**
- `socketRef` — holds the Socket.io client instance
- `audioRef` — holds the `<HTMLAudioElement>` from the DOM

**State (triggers re-renders):**
- `currentSong` — what is playing right now
- `queue` — upcoming songs
- `downloadStatuses` — per-URL download progress/errors
- `needsInteraction` — true if autoplay was blocked by the browser
- `connected` — whether the socket is connected

**On mount (`useEffect`):**
1. Creates a `socket.io-client` connection to `http://localhost:4000`.
2. On `connect`, emits `joinRoom` to get the initial room state.
3. Registers handlers for all five server events (see below).
4. On unmount, disconnects the socket.

**Event: `roomState`**
Fires once when the client first joins. Sets `currentSong` and `queue`.
If a song is currently playing, sets `audio.src`, seeks to `elapsed`,
and calls `.play()`. If `.play()` throws (autoplay blocked), sets
`needsInteraction = true`.

**Event: `songStarted`**
Fires when the server advances to the next song. Sets `currentSong` and
`queue`. Resets `audio.currentTime = 0` and plays from the beginning.

**Event: `queueUpdated`**
Fires when anyone adds a song. Updates `currentSong` and `queue` in state
(does not touch audio — the song didn't change, only the queue did).

**Event: `downloadStatus`**
Upserts into the `downloadStatuses` array by matching on `url`.
If status is `done` or `error`, schedules removal of that entry after 3 seconds.

**`addToQueue(url)`**
Emits `addToQueue` to the server. The URL may be a direct link or a
`ytsearch1:...` prefixed search query (transformed in `SearchBox`).

### `components/NowPlaying.tsx`

Receives `currentSong: QueueItem | null`.

When a song is playing: renders the title and four animated bars using CSS
`@keyframes wave` (defined in `global.css`). Each bar has a different
`animation-delay` creating a staggered wave effect.

When nothing is playing: renders "Nothing playing yet."

Does not show duration, timestamp, or any playback controls.

---

### `components/Queue.tsx`

Receives `queue: QueueItem[]`.

Renders a numbered `<ol>` of upcoming song titles.
Empty state: "Queue is empty — add a song below."

---

### `components/SearchBox.tsx`

Receives `downloadStatuses` and `onAdd` callback.

**Input logic:**
- If input starts with `http` → send as-is (direct URL)
- Otherwise → prefix with `ytsearch1:` so yt-dlp searches YouTube

**Status indicators:**
- `downloading` → pulsing violet dot + percentage
- `done` → green dot + "Added to queue" (disappears after 3s)
- `error` → red dot + error message (disappears after 3s)

---

### `app/page.tsx`

The single page. Calls `useRoom()` and passes the returned values down
to each component. Also renders:
- A connection status badge (green "connected" / grey "connecting…")
- A "Click to start listening" button when `needsInteraction` is true
- The hidden `<audio ref={audioRef} />` element

The `<audio>` element is hidden and has no browser controls. It is driven
entirely by the `useRoom` hook.

## Full Data Flow: Adding a Song

This is the complete path from user action to music playing:

```
1. User types "lofi hip hop" into SearchBox and hits Add
        │
        ▼
2. SearchBox calls onAdd("ytsearch1:lofi hip hop")
        │
        ▼
3. useRoom.addToQueue emits socket event addToQueue { url }
        │
        ▼ (crosses network to backend)
4. RoomGateway.handleAddToQueue receives it
        │
        ├── broadcasts downloadStatus { status: 'downloading', progress: 0 }
        │       └── all clients show a pulsing indicator
        │
        ▼
5. QueueService.downloadAndEnqueue spawns yt-dlp
        │
        ├── as yt-dlp runs, broadcasts downloadStatus { progress: 42.3 }
        │
        ▼
6. yt-dlp finishes → QueueItem created, .mp3 saved to audio-cache/
        │
        ├── broadcasts downloadStatus { status: 'done', item }
        │
        ▼
7. RoomGateway calls RoomService.enqueue(item)
        │
        ├── if nothing playing → RoomService.advanceSong() runs immediately
        │       ├── sets currentSong, startedAt = Date.now()
        │       ├── emits internal event room.songAdvanced
        │       └── schedules next advanceSong after duration seconds
        │
        └── broadcasts queueUpdated to all clients
        │
        ▼ (if song just started)
8. RoomGateway.handleSongAdvanced receives room.songAdvanced
        │
        └── broadcasts songStarted { currentSong, startedAt, queue }
        │
        ▼ (crosses network to all browsers)
9. useRoom on('songStarted') in every connected browser
        │
        ├── sets audio.src = /audio/{fileId}.mp3
        ├── sets audio.currentTime = 0
        └── calls audio.play()
        │
        ▼
10. Browser fetches /audio/{fileId}.mp3 from NestJS (range requests)
        └── music plays in sync across all clients
```

## The Sync Problem — How "Join at 1:30" Works

When a client joins mid-song, the server calculates:

```ts
elapsed = (Date.now() - state.startedAt) / 1000  // in seconds
```

This is sent inside the `roomState` event. The client then:

```ts
audio.src = `/audio/${currentSong.fileId}.mp3`
audio.currentTime = elapsed   // seek to 1:30
audio.play()
```

Because all clients stream the same file and seek to the same offset
derived from the same server clock, they play in sync. There is no
peer-to-peer coordination. The server's `Date.now()` is the single
source of truth.

## What This App Deliberately Does NOT Do

| Missing feature       | Reason                                      |
|-----------------------|---------------------------------------------|
| Play / pause / skip   | Music flows like a radio — no controls      |
| User login / accounts | Not needed for a small friend group         |
| Database              | State resets on restart — intentional       |
| Multiple rooms        | One room only in v1                         |
| Next.js API routes    | All stateful logic lives in NestJS only     |
| Video playback        | Audio only                                  |
